# ModelWatch — Claude Code Context

## What this is
Continuous behavioral drift monitoring for LLM-powered applications. Users
define behavioral specs (input → expected output rules), ModelWatch runs them
on a schedule against the user's chosen LLM endpoint, compares outputs against
a baseline, and alerts when significant drift is detected. Catches *provider-side*
model updates that silently break customer applications — what DeepEval/Braintrust/
LangSmith can't catch because they only run on user trigger.

Customers bring their own OpenAI/Anthropic API key (zero LLM COGS for us).

## Architecture
```
app/
  config.py            — pydantic-settings (env vars, tier limits)
  main.py              — FastAPI app, lifespan starts scheduler + creates tables
  models/
    database.py        — lazy async SQLAlchemy engine/session (sqlite-friendly)
    schemas.py         — ORM: Workspace, ApiKey, Endpoint, Spec, Run, DriftEvent
    api_models.py      — Pydantic request/response models
  middleware/auth.py   — API-key auth (mw_<token> Bearer; NOT Clerk)
  routers/
    auth.py            — /api/auth/signup + /api/auth/rotate-key
    workspaces.py      — /api/workspaces/me + /me/api-keys (LLM provider keys)
    endpoints.py       — /api/endpoints CRUD
    specs.py           — /api/specs CRUD + /run + /runs + /drift-events + /reset-baseline
    dashboard.py       — /api/dashboard/health + /drift-trends + /recent-events
    billing.py         — /api/billing/checkout + /webhook (Stripe)
    badge.py           — /api/badge/{workspace_id}.svg (shields.io-compat SVG)
  services/
    encryption.py      — Fernet for customer LLM keys at rest
    llm_executor.py    — async OpenAI + Anthropic SDK wrappers (uses customer keys)
    embedding.py       — text-embedding-3-small via customer's OpenAI key
    diff_engine.py     — format / length / semantic / refusal / contains diff
    drift_detector.py  — orchestrator: run spec → diff → maybe create DriftEvent
    scheduler.py       — APScheduler hourly/daily/weekly + monthly counter reset
    alerts.py          — SendGrid email + Slack incoming-webhook
    billing.py         — Stripe checkout + webhook handlers + plan limits
tests/                 — 53 tests, all passing
deploy.sh              — full provisioning (Stripe → Railway → Cloudflare)
Dockerfile             — python:3.12-slim, runs `python -m app.main`
railway.toml           — startCommand uses `python -m app.main` (no shell)
.env.example           — env var documentation
.encryption_key        — generated by deploy.sh, gitignored
.env.railway           — populated by deploy.sh during run, gitignored
```

## Critical rules
- API keys are stored Fernet-encrypted; never log or return plaintext
- Per Brett's autonomous-only rule: NO Clerk, NO manual dashboard onboarding.
  Auth is `mw_<token>` API key, minted at /api/auth/signup, emailed via SendGrid.
- Every endpoint check goes through plan-limit enforcement (`check_limit` raises 403)
- Diff scoring weights: semantic 0.35, format 0.20, refusal 0.20, length 0.15, contains 0.10
- Severity buckets: <0.05 none, <0.15 low, <0.35 medium, <0.6 high, ≥0.6 critical
- Railway startCommand uses `python -m app.main` (no shell, so $PORT must be read via os.getenv — already done in main.py)
- `RAILWAY_TOKEN` from `.deploy-secrets.env` is account-scoped; deploy.sh copies it to `RAILWAY_API_TOKEN` and unsets `RAILWAY_TOKEN` (CLI rejects when both are set)

## Deploy — exact steps for Brett's Mac

### Prereqs (one-time)
- `brew install railway`
- `pip install stripe cryptography` (deploy.sh uses both)
- `.deploy-secrets.env` exists at workspace parent (it does — verified)

### Run
```bash
cd "/Users/bretthalverson/Projects/agentic-builds/Build Prompts from OpenClaw/modelwatch"
bash deploy.sh
```

### What deploy.sh does (idempotent, re-run-safe)
1. **Encryption key** — generates Fernet key, saves to `.encryption_key`
2. **Stripe** — `Product.search` by metadata.slug='modelwatch'; creates 3 prices via lookup_keys
3. **Railway** — `railway init`, `railway add --database postgres`, creates `backend` service with all env vars including `DATABASE_URL=${{Postgres.DATABASE_URL}}`
4. **Deploy** — `railway up --service backend --detach`, captures auto-generated `*.up.railway.app` URL
5. **Cloudflare DNS** — looks up zone for `modelwatch.app`. If absent, prints CF setup instructions and exits cleanly (Railway URL still works). If present, upserts CNAMEs for `api.`, apex, `www.`
6. **Custom domain** — `railway domain api.modelwatch.app --service backend` etc.

### Manual touches (one-time, after first deploy)

1. **Cloudflare zone for modelwatch.app**
   - Dashboard → Add a Site → `modelwatch.app`
   - Note the two assigned nameservers
   - Update nameservers at the registrar where modelwatch.app was purchased
   - Wait for activation, then `bash deploy.sh` again — step 5 will wire DNS

2. **Stripe webhook**
   - Stripe Dashboard → Developers → Webhooks → Add endpoint
   - URL: `https://api.modelwatch.app/api/billing/webhook` (or the Railway URL pre-DNS)
   - Events: `checkout.session.completed`, `customer.subscription.deleted`, `invoice.payment_failed`
   - Copy the new signing secret. If it differs from `STRIPE_WEBHOOK_SECRET` in `.deploy-secrets.env`:
     ```bash
     railway variables --service backend --set STRIPE_WEBHOOK_SECRET=whsec_NEW
     ```
